# Picker 与搜索编辑协同工作流设计

## 1. 背景

当前产品把高频快选放在 `Picker`，把完整搜索和编辑放在 `Manager`。这套拆分在早期有助于控制复杂度，但已经暴露出明确问题：

- 用户需要在 `Picker` 与 `Manager` 之间频繁切换，进入搜索和编辑的操作步骤偏多。
- `Picker` 与 `Manager` 的职责分离过于生硬，高频链路会被窗口切换打断。
- 现有 `Manager` 更像完整资料库，而不是围绕当前上下文快速处理内容的工作窗。

本轮设计的目标不是把所有能力塞回 `Picker`，而是减少窗口来回跳转，让搜索和编辑成为围绕 `Picker` 的顺滑延伸。

## 2. 设计目标

本轮需要同时满足以下目标：

1. 保持 `Picker` 主外观和主流程基本不动。
2. 降低从快选进入搜索和编辑的成本。
3. 搜索和编辑应成为一条连续链路，而不是再打开一个重资料库页面。
4. `Picker` 仍以“不抢焦点优先”为第一原则。
5. `Manager` 不再承担高频搜索与编辑，可退化为设置页或低频入口。

## 3. 已确认约束

本轮已经确认的产品约束如下：

- `Picker` 必须保留。
- 搜索和编辑需要更流畅进入，但不能破坏 `Picker` 的主视觉和主流程。
- 允许从 `Picker` 跳到专用窗口，但不是简单再开一个传统 `Manager`。
- 编辑高频能力优先支持文本项。
- 高频路径内需要支持文本编辑、收藏、删除。
- 唤起 `Picker` 时仍以尽量不抢焦点为第一原则。
- 交互偏好为键鼠混合：键盘快选依旧重要，鼠标点搜索或编辑也必须顺手。

## 4. 方案结论

采用“协同双窗”方案：

- `Picker` 保持现有快选角色，不承载完整搜索输入和完整编辑界面。
- 新增一个专用“搜索/编辑工作窗”，承接深操作。
- `Picker` 只新增上下文跳转能力，将当前会话和当前条目交给工作窗。
- 全局也允许直接进入工作窗的搜索态。
- 当前 `Manager` 退化为设置页或低频配置入口，不再承担高频搜索和编辑。

该方案的核心不是合并窗口，而是让两个窗口共享同一条会话链路，避免用户在两个互不相关的界面之间重新开始。

## 5. 窗口职责

### 5.1 Picker

`Picker` 继续承担以下职责：

- 展示近期记录并支持高频快选。
- 保持现有主外观、主节奏和回贴主链路。
- 提供两类跳转能力：
  - 对当前选中项发起“编辑此项”。
  - 发起“进入搜索”或“基于当前项搜索”。

`Picker` 不承担以下职责：

- 常驻搜索输入框。
- 常驻文本编辑区。
- 复杂资料库管理功能。

### 5.2 搜索/编辑工作窗

工作窗承担以下职责：

- 搜索文本记录。
- 对文本记录进行编辑。
- 对记录执行收藏、取消收藏、删除、回贴。
- 在同一窗口内连续完成“搜索 -> 选择 -> 编辑 -> 回贴”。

工作窗的定位不是资料库，而是围绕当前上下文快速处理内容的工作台。

### 5.3 设置页

现有 `Manager` 调整为设置页或低频入口，主要负责：

- 快捷键配置。
- 监听开关。
- 排除应用。
- 主题与历史上限等低频设置。

## 6. 会话模型

本轮需要引入明确的“会话”概念，以保证 `Picker` 与工作窗的上下文连续。

### 6.1 会话类型

定义两类会话：

- `picker_session`
- `workbench_session`

### 6.2 会话基础信息

每个会话至少需要保存以下信息：

- 会话来源类型。
- 会话开始前的目标窗口句柄。
- 是否由 `Picker` 跳转而来。
- 当前选中条目 id。
- 初始搜索意图。
- 是否已经完成回贴。

### 6.3 回贴目标规则

回贴目标统一定义为“当前会话开始前的外部目标窗口”。

具体规则如下：

- 从 `Picker` 链路进入工作窗：回贴到打开 `Picker` 前的目标窗口。
- 全局直接进入工作窗：回贴到打开工作窗前的目标窗口。

这条规则用于避免把回贴目标错误写死为 `Picker` 前的窗口。

## 7. 入口与交互流

### 7.1 Picker 主流程

`Picker` 的主流程保持不变：

- 上下移动选择条目。
- 回车或快捷键直接回贴。
- 数字快捷键快速选择。

### 7.2 从 Picker 进入编辑

当用户在 `Picker` 中选中某一项时，可触发“编辑此项”动作。

建议同时提供两类入口：

- 键盘入口，例如 `Ctrl+E`。
- 鼠标入口，例如选中态或悬停态下的轻量动作按钮。

触发后：

1. 隐藏 `Picker`。
2. 打开工作窗。
3. 带当前条目 id 进入工作窗的编辑态。
4. 左侧仍保留相关记录列表，方便顺手切换到别的记录。

### 7.3 从 Picker 进入搜索

`Picker` 可触发“进入搜索”动作，但不在自身承载搜索框。

建议同时提供两类入口：

- 键盘入口，例如 `Ctrl+F`。
- 鼠标入口，例如一个轻量的“搜索更多”操作。

触发后：

1. 隐藏 `Picker`。
2. 打开工作窗搜索态。
3. 搜索框自动聚焦。
4. 左侧显示最近项或搜索结果。

### 7.4 基于当前项搜索

在 `Picker` 中对某一项触发“基于当前项搜索”时：

- 工作窗带着初始关键词进入搜索态。
- 初始关键词来源于该项的可搜索摘要，而不是整段全文。
- 建议优先使用预览文本、标题或受控截断文本。

### 7.5 全局直接进入搜索

除 `Picker` 跳转外，工作窗还应支持独立全局快捷键。

这条链路适用于用户一开始就明确要搜索或编辑，而不是先快选。

触发后：

1. 直接打开工作窗搜索态。
2. 搜索框自动聚焦。
3. 默认不预填关键词。
4. 用户在结果中选择条目后进入编辑态。

## 8. 搜索关键词策略

搜索初始关键词必须来自明确来源，不允许隐式猜测。

规则如下：

- 从 `Picker` 普通进入搜索：默认不预填关键词。
- 从 `Picker` 发起“基于当前项搜索”：才带初始关键词。
- 全局直接进入工作窗搜索：默认空关键词并自动聚焦输入框。

本轮不引入“读取前台应用选中文本作为初始搜索词”的能力，以避免额外平台复杂度和兼容性风险。

## 9. 工作窗布局

工作窗采用聚焦高频处理的双栏结构，而不是当前资料库式三栏结构。

### 9.1 顶部操作带

顶部只保留必要信息和少量高频动作：

- 当前入口来源提示。
- 搜索框。
- 关闭动作。
- 与回贴目标相关的轻量提示。

### 9.2 左侧结果栏

左侧负责“找”和“切换条目”：

- 从条目编辑进入时，默认高亮当前条目并展示邻近记录。
- 从搜索进入时，展示搜索结果。
- 列表项展示最小必要信息：预览、时间、类型、收藏状态。
- 支持键盘上下切换和鼠标点选。

### 9.3 右侧编辑栏

右侧负责“处理当前条目”：

- 文本项：显示正文编辑区和高频操作按钮。
- 非文本项：不提供文本编辑，仅展示元信息与允许的操作。

高频操作包括：

- 保存文本。
- 收藏 / 取消收藏。
- 删除。
- 回贴。

## 10. 工作窗状态

工作窗内部只保留三个清晰状态：

### 10.1 搜索态

- 搜索框聚焦。
- 左侧显示结果列表。
- 右侧在未选中时显示占位说明。

### 10.2 编辑态

- 右侧显示当前条目的完整内容和操作。
- 左侧仍可继续切换别的条目。
- 保存后留在当前状态。

### 10.3 空态

- 无结果或尚未选中条目时显示。
- 仅提供轻量提示，不堆叠复杂说明。

## 11. 收尾与回退规则

### 11.1 保存

- 纯编辑保存后，工作窗保持打开。
- 不自动关闭，不自动返回 `Picker`。

### 11.2 回贴

“回贴”定义为：将当前条目内容粘贴到当前会话开始前的外部目标窗口。

回贴成功后：

1. 关闭工作窗。
2. 如果当前链路来自 `Picker`，同时结束 `Picker` 会话。
3. 恢复目标窗口。
4. 执行粘贴。

回贴是当前会话的终点动作，而不是普通编辑动作。

### 11.3 关闭工作窗

- 如果当前入口来自 `Picker`，关闭后默认回到原 `Picker` 上下文。
- 回到 `Picker` 时应尽量恢复原选中项和原列表位置。
- 如果是全局直接进入工作窗，关闭后直接结束当前工作窗会话。

### 11.4 删除

- 删除当前条目后，如果左侧还有下一项，则自动切到下一项。
- 如果没有可切换项，则进入空态或结果态。
- 删除后不自动返回 `Picker`。

## 12. 错误处理与边界行为

### 12.1 工作窗打开失败

- 从 `Picker` 发起跳转但工作窗未成功打开时，`Picker` 不应丢失。
- 系统需要恢复或保持 `Picker` 可用状态，并提示打开失败。

### 12.2 目标窗口失效

- 回贴前若发现目标窗口不存在或不可恢复，应提示目标窗口不可用。
- 此时不自动关闭工作窗，避免用户失去当前编辑内容。

### 12.3 条目失效

- 若从 `Picker` 带入的条目在工作窗打开前已删除或失效，工作窗应退回空态或结果态。
- 同时提示原条目已不存在。

### 12.4 非文本条目

- 图片和文件条目允许查看、收藏、删除、回贴。
- 不进入文本编辑区，不伪造不可用编辑态。

## 13. 对现有代码结构的影响

本轮产品形态采用新工作流，但工程实现应尽量复用现有能力。

建议复用的现有部分：

- 搜索查询逻辑。
- 条目详情查询逻辑。
- 文本更新、收藏、删除、回贴相关命令和 mutation。
- 现有窗口协调能力。

建议重构的部分：

- 将 `Manager` 当前“资料库 + 设置”混合职责拆开。
- 单独建立工作窗的视图状态与会话状态。
- 扩展窗口协调层，支持 `Picker` 与工作窗之间的上下文传递和回退。

## 14. 验证场景

后续实现至少需要验证以下场景：

1. `Picker -> 编辑此项 -> 保存 -> 关闭工作窗 -> 回到 Picker`
2. `Picker -> 搜索 -> 选中结果 -> 编辑 -> 回贴`
3. `Picker -> 基于当前项搜索 -> 结果切换 -> 编辑 -> 保存`
4. `全局直接进入搜索 -> 搜索 -> 选中 -> 编辑 -> 回贴`
5. `Picker -> 工作窗打开失败 -> Picker 仍可继续使用`
6. `工作窗内删除当前项 -> 自动切换下一项或进入空态`
7. `非文本条目进入工作窗 -> 不显示文本编辑区`
8. `目标窗口失效 -> 回贴失败但工作窗保留`

## 15. 非目标

本轮明确不做以下事项：

- 在 `Picker` 内加入常驻搜索框。
- 在 `Picker` 内加入常驻编辑区。
- 读取外部应用选中文本作为默认搜索词。
- 为图片或文件引入复杂编辑能力。
- 保留现有 `Manager` 作为高频资料库主入口。

## 16. 结论

本轮设计通过“保留 `Picker`、新增工作窗、共享会话上下文”的方式，解决高频搜索和编辑需要频繁切换窗口的问题。

它保住了 `Picker` 的快选特性和非焦点优先原则，同时让搜索和编辑进入一条更短、更连续的链路。后续实现应优先保证会话目标窗口、上下文恢复和回贴闭环的正确性，再推进界面替换与交互细节收口。
