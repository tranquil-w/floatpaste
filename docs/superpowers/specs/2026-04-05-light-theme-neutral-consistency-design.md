# 浅色主题跨设备中性一致性设计

## 1. 背景

当前仓库的浅色主题并不是中性白灰基底，而是已经主动调整为暖灰纸感。相关实现位于 `src/index.css` 的浅色 `:root` token，设计背景也已记录在 `docs/superpowers/specs/2026-04-01-warm-neutral-theme-design.md`。

这带来了一个新的问题：同样一套浅色 token，在不同屏幕上会被不同程度地放大。

- 在偏冷或校色较中性的屏幕上，当前主题可能只表现为“微暖”
- 在白点偏暖、面板色彩更艳、或系统开启夜间/护眼模式的屏幕上，当前主题会更明显地偏黄

本次设计的核心判断是：

1. 这更像是“显示设备差异放大了现有暖色倾向”，而不是 Tauri 本身额外引入的色偏
2. 应用内新增“色温调节”并不能真正解决跨设备一致性，只能改变本应用 token 的主观冷暖感

因此，本次工作的目标不是继续强化暖调风格，而是让默认浅色主题在不同屏幕上更稳定、更中性，避免出现“同一主题在另一台电脑上忽然变得很暖”的体验落差。

## 2. 目标

### 2.1 主要目标

- 让默认浅色主题在不同屏幕上的观感尽量一致
- 降低暖屏设备把浅色界面放大成“明显暖黄”的风险
- 保留清晰、专业、稳定的产品气质

### 2.2 非目标

- 不尝试做真正意义上的显示器校色
- 不引入自动判断设备冷暖并动态修正的逻辑
- 不在本次迭代中新增“色温滑杆”或连续调节系统
- 不重做整套主题系统
- 不改动深色主题

## 3. 方案结论

本次采用：

`A1：直接把默认浅色 token 回调到更中性的基底`

不采用：

- `A2：默认改中性，同时保留经典暖调浅色预设`
- `A3：尝试做自动校正逻辑`
- 新增应用内“色温调节”系统

原因如下：

1. `A1` 最直接命中问题根源。当前风险来自默认浅色主题本身已经带暖调，先把默认值校正回更稳的位置，收益最大，复杂度最低。
2. `A2` 会把问题从“默认值修正”扩展到“主题预设管理”，需要同步调整设置 UI、存储结构与说明文案，不适合作为第一步。
3. `A3` 缺乏可靠的设备色彩信息输入，容易做成不可预测的“智能变色”。
4. “色温调节”并不能修复屏幕白点、面板校色、系统夜间模式等客观差异，只能在应用层挪动视觉重心。它更适合作为主观偏好功能，而不是跨设备一致性的第一层解决方案。

## 4. 设计原则

### 4.1 默认主题先追求稳定，再追求风格个性

浅色主题首先应该在不同设备上都成立，而不是只在当前开发机器上看起来舒服。默认主题不应依赖“这块屏幕刚好偏冷”来达成平衡。

### 4.2 只校正基底层，不重做语义层

本次优先修改浅色模式中的中性色阶与浅层背景色，让大面积背景、卡片底、hover/selected 浅底先回到中性。主强调蓝、状态色、深色主题暂不调整。

### 4.3 不把显示器问题伪装成应用设置

如果默认值本身不稳定，先修默认值，而不是把问题交给用户通过滑杆自己拧回去。用户设置只应该承接偏好，不应该承担补救责任。

## 5. 具体改动范围

### 5.1 修改范围

只调整浅色 `:root` 下与整体色温感直接相关的 token：

- `--pg-neutral-0` 至 `--pg-neutral-7`
- 由中性色阶映射得到的浅色 canvas 观感
- 浅色 `--pg-accent-subtle`
- 浅色 `--pg-shadow-color`

预期效果：

- 页面大底色更接近中性浅灰白
- 卡片、输入框、面板底色不再明显带米黄感
- hover/selected 的浅色承托层回到更稳定的低强度中性表达
- 阴影不再带偏棕感

### 5.2 明确保持不变的部分

- `--pg-accent-fg` 等主强调蓝不变
- 成功、警告、危险、完成等语义色不变
- 深色主题 token 不变
- `themeMode`、桥接层、Tauri 命令层不变
- 组件层 className 原则上不逐个修改，优先通过 token 生效

## 6. Token 调整策略

### 6.1 浅色 neutral 收回到“中性但不冰冷”

当前浅色 neutral 已经明显向暖灰纸感倾斜。本次不直接回到刺眼纯白，也不继续维持纸黄感，而是回到更接近 Primer 浅色体系的中性灰白区间。

方向如下：

1. `neutral-0 ~ neutral-3` 负责整体底色印象，应最优先去暖化
2. `neutral-4 ~ neutral-7` 负责边框、分隔与控件承托层，应同步收中性，避免界面整体继续泛黄
3. 保留极轻微暖意可以接受，但只能停留在“几乎不可感知”的程度，不能成为浅色主题第一印象

### 6.2 accent-subtle 从暖米色回到更稳定的浅强调色

当前浅色 `--pg-accent-subtle` 为暖色值，这会让选中态、hover 态和局部高亮一起参与整体升温。本次将其改回更中性的浅强调表达，优先考虑：

- 回归基于蓝色的极浅 subtle 层，或
- 使用低强度冷中性浅底

原则是：

- 保持交互反馈清晰
- 不让浅交互底色把整个页面烘成暖调

### 6.3 shadow-color 同步回中性

如果底色已经回中性，而阴影仍偏棕，会继续给界面留下“泛暖”的综合色偏。因此阴影通道也应同步收回中性灰。

## 7. 实现边界

### 7.1 本次不新增设置项

设置页不新增“色温”“暖度”“冷暖调节”等入口。原因：

1. 当前优先级是把默认主题修正到更稳定
2. 新设置会扩大用户决策成本
3. 连续滑杆很容易制造“数值可调但结果不可控”的体验

如果未来仍有明确需求，更合理的后续演进是：

- 在默认中性浅色稳定后，再考虑新增“暖调浅色”作为独立预设
- 不建议新增连续色温滑杆作为基础能力

### 7.2 本次不引入设备自适应逻辑

不根据设备、显示器信息、系统状态做自动冷暖修正。当前运行环境难以可靠感知真实色彩链路，强行自适应会带来不可预测性。

## 8. 验证方案

### 8.1 构建验证

前端修改后至少执行：

`./scripts/win-pnpm build`

### 8.2 视觉验证

至少检查以下场景的浅色模式：

- 设置页
- Picker 列表页
- 编辑页

重点观察：

- 页面大面积背景是否已经从纸黄感回到中性浅灰白
- 卡片、输入框、面板底色是否更稳定
- hover / selected 的浅底是否仍有清晰反馈，但不再整体发暖
- 蓝色主强调是否仍然明确
- 阴影、边框、层次是否保持清晰

### 8.3 跨设备复看

如果条件允许，至少在另一台更偏暖的屏幕上复看一次。预期结果不是“完全一致”，而是：

- 当前机器上：不再以暖纸感作为第一印象
- 偏暖机器上：不会被放大成明显暖色主题

## 9. 验收标准

- 默认浅色主题的第一印象为中性、清晰、稳定
- 不同屏幕之间的观感差异仍然存在，但不再从“微暖”扩大到“明显暖黄”
- 深色主题视觉保持不变
- 主交互强调、可读性、层次感不下降
- 不引入新的设置复杂度

## 10. 风险与后续

### 10.1 主要风险

最主要的风险不是功能错误，而是去暖后可能显得过于普通、偏冷，失去当前“纸感”特色。

### 10.2 风险处理

本次通过“中性但不冰冷”的策略控制风险，而不是彻底回到纯白冷灰。也就是说：

- 先把明显暖感拿掉
- 保留极轻微、几乎不可感知的温和感

### 10.3 后续演进

如果默认中性浅色上线后，仍有明确的暖调偏好需求，可在后续单独设计：

- `标准浅色（默认）`
- `暖调浅色（可选）`

但这应作为主题预设问题单独设计，不与本次“跨设备一致性修正”混在一起。

## 11. 实施摘要

本次迭代将默认浅色主题从“暖灰纸感基底”收回到“中性浅色基底”，通过校正浅色 neutral、`accent-subtle` 与 `shadow-color`，降低不同屏幕对白色温度差异的放大效应。

这不是 Tauri 层面的渲染修复，而是一次有明确边界的主题 token 校正。目标是让 floatpaste 的默认浅色主题在更多设备上都显得稳定、克制、专业，而不是要求用户通过应用内色温功能自行补偿显示器差异。
