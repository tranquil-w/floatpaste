# Search 内联展开与自适应窗口布局设计

**日期：** 2026-04-05  
**范围：** `Search` 窗口布局与窗口行为重构

## 1. 背景

当前 `Search` 窗口采用“结果列表 + 底部摘要区”的双层结构。

这带来两个明显问题：

1. 用户需要先在列表中找到目标，再把视线拉到底部确认摘要和操作
2. 当前项的判断与处理不发生在同一视觉位置，节奏被拆成了两段

对 `Search` 这种承接高频“找 -> 判断 -> 处理”链路的窗口来说，这个结构过于分离。  
用户不应该在一个本来偏瞬时的工具窗里反复上下移动视线，也不应该为了执行 `粘贴 / 编辑 / 收藏`，先完成一次“从结果区跳到摘要区”的认知切换。

此外，当前窗口尺寸也偏像资料库页面：

- 默认宽度偏大
- 高度固定感较强
- 结果少时仍占用较大面积
- 高度与结果数量没有形成自然呼应

这与本轮希望强化的“更短、更快、更像搜索处理窗”的方向不一致。

## 2. 目标

### 2.1 主要目标

- 用“当前项内联展开”替代底部固定摘要区
- 让搜索、判断、处理保持在同一条垂直视线上完成
- 让窗口尺寸更贴近结果数量，而不是始终维持较重的资料库体量
- 保持 `Search` 仍然以列表为主，而不是演变成另一个详情页

### 2.2 非目标

- 不把 `Search` 改造成完整编辑器
- 不新增右侧详情面板
- 不恢复底部固定摘要区
- 不在本次迭代中重做筛选逻辑
- 不修改 `Picker` 和 `Editor` 的现有交互语义

## 3. 用户已确认的方向

本次设计已确认以下偏好：

- 优先问题是：当前“列表 + 底部摘要区”不顺，需要重构布局
- 布局方向采用：`当前项在列表内联展开`
- 展开强度采用：`中等内联展开`
- 其他项保持正常高度，不额外压缩
- 窗口默认宽度需要减小
- 窗口高度不允许手动调整
- 窗口高度跟随列表项数量自动变化
- 到达最大高度后，窗口固定，结果列表内部滚动

## 4. 方案结论

采用：

`A2：中等内联展开 + 自适应高度窗口`

不采用：

- `A1：轻展开`
- `A3：重详情式内联`
- `B 类：保留底部摘要区，仅调整比例`
- `C 类：改成右侧详情区`

选择 `A2` 的原因：

1. `A1` 只会把底部摘要区“搬到列表里”，但仍不足以真正建立新的处理节奏
2. `A3` 容易让 `Search` 变成半详情页，损害搜索结果列表的轻量感
3. `A2` 能在保持列表主导地位的前提下，让“当前项”承担完整的判断与主操作角色
4. 与自动高度结合后，窗口整体会更像瞬时搜索面板，而不是资料库页

## 5. 设计原则

### 5.1 结果与操作必须共处一处

用户不应该在搜索结果和操作区之间来回跳。  
当前项的展开区必须直接依附于当前结果项，让判断与执行发生在同一个视觉块中。

### 5.2 展开区是“处理槽”，不是“详情页”

展开区的目标是帮助快速判断和执行，不是鼓励长时间停留。  
因此它应包含足够信息，但不能膨胀成第二层页面。

### 5.3 列表仍然是 Search 的主骨架

即使当前项展开，用户也应该持续感觉自己仍在操作“搜索结果列表”，而不是离开列表进入详情视图。

### 5.4 窗口尺寸应响应任务密度

结果少时，窗口应该更短、更贴身。  
结果多时，再通过内部滚动承接密度，而不是一开始就占据较大体量。

## 6. 具体改动

## 6.1 取消底部固定摘要区

移除当前底部固定摘要区，不再使用一个独立横向区域来展示：

- 选中项摘要
- 文本/非文本说明
- 粘贴 / 编辑 / 收藏按钮

这些内容全部迁移到当前选中项的内联展开区。

预期结果：

- 结果列表与操作不再分裂成上下两层
- 当前项成为唯一需要被关注的结果块

## 6.2 当前项内联展开

当前选中项在列表内原位展开，展开强度为“中等”。

展开区包含 4 类内容：

1. 更完整的摘要  
   比基础列表项更多，但不是全文编辑器

2. 必要上下文  
   类型、来源、时间等确认信息，继续保留为辅助层

3. 主操作  
   `粘贴`、`编辑`、`收藏 / 取消收藏`

4. 状态反馈  
   详情加载中、非文本提示、空数据提示等

预期结果：

- 用户在同一处完成“看是不是这条”和“下一步操作”
- 当前项更像一个处理槽，而不是一个只是被高亮的列表行

## 6.3 其他项保持正常高度

除当前项外，其他结果项保持正常列表高度，不额外压缩，也不做大幅退场。

原因：

- 本轮目标是内联展开，而不是做“聚焦模式”
- 保持其他项稳定高度有助于快速连续切换
- 可以避免展开项频繁跳动时，列表节奏显得过于激进

预期结果：

- 展开项突出，但列表整体仍稳定
- 用户切换上下项时，心智模型保持为“在结果列表里移动”

## 6.4 默认宽度减小

当前窗口默认宽度偏大，视觉上更像资料库页。  
本次将默认宽度收窄，目标是让窗口更像“当前任务搜索处理窗”。

设计要求：

- 默认宽度明显小于现状
- 仍需保证搜索框、结果摘要和内联操作不拥挤
- 不追求极窄，而是从“资料库宽度”退回“处理窗宽度”

预期结果：

- 搜索窗口整体更轻
- 展开项在更紧凑的宽度下仍保持清楚，但不再像大面板

## 6.5 高度跟随结果数量自动变化

窗口高度由结果数量驱动，而不是固定为较高尺寸。

规则如下：

- 结果很少时，窗口高度收缩
- 随结果数量增加，窗口逐步变高
- 达到最大高度后停止增长
- 超出部分由列表内部滚动承接

预期结果：

- 少量结果时，窗口更贴身
- 多量结果时，仍保留列表浏览能力
- 不会因为默认高度过大而显得沉重

## 6.6 禁止手动调整高度

本次明确不允许用户手动调整窗口高度。

原因：

- 既然高度已经随结果数量自动变化，再保留手动调高/调低会造成两套尺寸逻辑并存
- `Search` 本轮目标是“瞬时处理窗”，不是长期停驻的自定义工作台
- 禁止高度调整可以让视觉和行为更稳定

设计要求：

- 窗口不允许高度 resize
- 若仍保留宽度 resize，需要单独评估；本轮至少明确去掉高度 resize 能力

## 6.7 列表内部滚动成为唯一溢出承接方式

当结果数量超出最大高度后：

- 窗口保持在最大高度
- 结果列表区域内部滚动
- 搜索输入区和窗口框架保持稳定

预期结果：

- 结果再多，窗口外框都不会继续膨胀
- 输入区与结果区关系稳定，不会因高度变化失去定位感

## 7. 不改动项

以下内容本次明确不动：

- 搜索关键词与筛选协议
- 收藏命令与回贴命令语义
- `Editor` 的会话逻辑
- `Picker` 到 `Search` 的入口语义
- 深色/浅色主题 token 体系

## 8. 影响文件

本次改动预计主要落在：

- `src/features/search/SearchShell.tsx`

可能涉及：

- `src/features/search/store.ts`
- `src/features/search/state.ts`
- `src/bridge/commands.ts`（若窗口尺寸控制命令需要补充）
- `src-tauri/src/services/window_coordinator.rs` 或相关窗口配置（若默认宽度/可调整方向在 Rust 层控制）

最终实现时应优先确认“窗口默认尺寸”和“是否允许高度 resize”当前由哪一层负责。

## 9. 验证策略

### 9.1 构建验证

前端改动后至少执行：

`./scripts/win-pnpm build`

### 9.2 布局验证

重点检查以下场景：

- 空结果
- 1-2 条结果
- 中等数量结果
- 达到最大高度后的长列表结果
- 文本项展开
- 非文本项展开

观察点：

- 是否已经取消底部固定摘要区
- 当前项展开后，判断与操作是否都发生在列表内
- 其他项是否仍保持正常高度
- 少量结果时窗口是否明显变短
- 达到最大高度后是否转为内部滚动

### 9.3 交互回归

至少手动确认：

- 搜索输入
- 上下键切换结果
- `Enter` 粘贴
- `Ctrl+Enter` 编辑
- 收藏切换
- 展开项切换时滚动是否仍稳定

## 10. 验收标准

- `Search` 不再使用底部固定摘要区
- 当前选中项在列表内联展开，承担摘要、上下文与主操作
- 展开区为中等强度，不演变成详情页
- 其他项保持正常高度
- 默认宽度比当前更窄
- 窗口高度随结果数量变化，并存在最大高度
- 到达最大高度后由列表内部滚动承接
- 用户在一条垂直视线上完成“搜索 -> 判断 -> 操作”

## 11. 实施摘要

本次迭代将 `Search` 从“列表 + 底部摘要区”的分层结构，重构为“单列表 + 当前项内联展开”的搜索处理窗，并同步把窗口尺寸改成更轻的自适应模型。

它的核心目标不是增加更多信息，而是把结果与操作重新放回同一处，并让窗口本身更像一个围绕当前任务临时浮现出来的工具。
