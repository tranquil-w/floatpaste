# Picker 结构化准星层级设计

**日期：** 2026-04-05  
**范围：** `Picker` 窗口视觉层级重构

## 1. 背景

当前 `Picker` 已经具备完整功能链路，但在视觉上仍偏“多条相似卡片并列”。  
对于这个窗口来说，最重要的不是展示更多信息，而是让用户在极短时间内确认：

- 当前选中的是哪一条
- 按下回车时会作用到哪一条

现状问题主要有三点：

1. 选中项虽然有高亮，但仍然和未选中项处在同一视觉层级带
2. 元信息、收藏状态、数字键提示与正文之间的主次关系还不够清楚
3. 顶部品牌与瞬时反馈的存在感，偶尔会与真正的操作目标竞争注意力

这与 FloatPaste 的核心体验目标并不完全一致。  
`Picker` 是一个高频、低停留时长、以“尽量不打断工作流”为前提的窗口，它需要更像一个专业工具里的“当前命中槽”，而不是一个会不断邀请用户浏览的内容面板。

## 2. 目标

### 2.1 主要目标

- 让用户打开 `Picker` 后，第一眼先看到“当前锁定项”
- 强化选中项与未选中项之间的层级差，而不是继续堆叠更多信息
- 保持冷静、专业、克制的工具气质
- 不通过展开内容来制造主次，而通过结构、字重、对比和信息收敛来建立主次

### 2.2 非目标

- 不新增交互行为
- 不修改 `Enter / Ctrl+Enter / 收藏 / 双击` 等现有语义
- 不恢复底部快捷键提示栏
- 不做选中项摘要展开
- 不引入强存在感动作条
- 不同时改动 `Search / Editor / Settings`

## 3. 用户已确认的方向

本次设计已确认以下偏好：

- `Picker` 优先解决“看得更准”，而不是“执行感更强”
- 方向采用“结构化准星”，不采用“执行强化型”
- 选中项**不展开内容**
- 整体风格偏**专业工具**，而不是偏品牌展示型
- 底部快捷键提示已删除，本次不恢复

## 4. 方案结论

采用：

`A2：结构化准星`

不采用：

- `A1：极简焦点框`
- `A3：命令行式准星`
- `B 类：带展开摘要的选中项`
- `C 类：强调执行动作的底部或内联动作条`

选择 `A2` 的原因：

1. `A1` 太保守，只会让当前高亮“稍微更明显”，很难真正拉开层级
2. `A3` 的工具感过强，会让 `Picker` 变得过硬，不利于图片、文件与文本混排
3. `A2` 可以在不展开内容、不增加动作负担的前提下，明确建立“当前命中项”与“候选列表”的关系

## 5. 设计原则

### 5.1 选中项必须成为唯一主角

打开 `Picker` 后，用户不应该先看到“一串列表”，而应该先感知到“当前锁定项”。  
这不依赖更大的面积，而依赖更清楚的层级组织。

### 5.2 用结构建立层级，不用展开制造层级

本次不通过展开更多文案、增加详情区域来强化选中态。  
层级差只通过以下元素建立：

- 边框与底色对比
- 字重和明度
- 数字键徽标的强调程度
- 元信息的收敛程度

### 5.3 未选中项应退后，但不能失去扫读能力

未选中项不是“被禁用”，只是“暂时不是目标”。  
它们仍需保持可读、可快速浏览，但不应与当前项竞争中心位置。

### 5.4 顶部区域只承担识别与状态，不承担视觉中心

`FloatPaste` 品牌识别需要保留，但不能压过列表本体。  
瞬时状态反馈也应更像系统状态提示，而不是会打断扫读的高频装饰。

## 6. 具体改动

### 6.1 选中项：从“高亮卡片”改为“锁定槽”

当前选中项保留单条卡片结构，但进一步明确成“当前锁定行”。

具体方向：

- 选中项边框比普通项更明确，但仍保持克制，不做厚描边
- 选中项底色与普通项形成更稳定的层级差
- 选中项正文使用略更强的字重与更干净的前景色
- 选中项内部的信息节奏更紧凑，让主文案成为明确重心

预期结果：

- 用户不需要逐项比对，就能直接识别当前项
- 选中态不依赖蓝色覆盖整张卡片，而是像“焦点被锁住”

### 6.2 数字键徽标：从辅助标签提升为命中标记

数字键提示在 `Picker` 中不只是元信息，它本身就是操作语义的一部分。  
因此它在选中项内应与“当前命中”建立更强关系。

具体方向：

- 未选中项的数字键徽标保持低存在感
- 选中项的数字键徽标在对比、底色与可读性上提升一级
- 徽标样式要更像“当前命中槽编号”，而不是普通角标

预期结果：

- 用户能更快建立“当前项 + 快捷数字”的映射
- 键盘选择语义被视觉化，而不是只停留在记忆层

### 6.3 正文与元信息：主次重新排序

当前正文、类型、来源、时间、收藏星标都能看到，但关系还不够收束。  
本次要把它们重新排列为：

1. 先看正文
2. 再确认上下文
3. 最后确认状态

具体方向：

- 正文优先级提升
- 元信息整体更细、更收敛、更像“确认层”
- 收藏状态仍可见，但不应抢过正文
- `type / source / time` 更强调扫描秩序，而不是并列平铺

预期结果：

- 用户先锁定内容，再补充判断来源与时间
- 单条记录内部不再像多个信息块并列争抢注意力

### 6.4 未选中项：统一退后半步

未选中项整体保留现有密度，但降低竞争力。

具体方向：

- 边框存在感降低
- 背景层更平
- 正文与元信息保持可扫读，但对比略退
- 收藏项可保留独立识别，但仍需服从整体层级

预期结果：

- 列表依旧可浏览
- 视觉重点不再分散到多条记录上

### 6.5 顶部区域：收敛品牌，稳定状态

顶部区域保留：

- 产品名
- 当前瞬时状态反馈

但调整目标是“退到辅助层”。

具体方向：

- 品牌识别缩回到工具识别位，而不是内容中心
- 瞬时反馈改为更稳定、低打扰的状态呈现
- 不使用会明显打断注意力的闪烁或过强强调

预期结果：

- 用户在打开窗口时，注意力先落到列表主区
- 顶部仍然清楚，但不再成为第一视觉落点

## 7. 不改动项

以下内容本次明确不动：

- Tooltip 逻辑
- 图片缩略图显示逻辑
- 收藏切换逻辑
- 打开编辑器逻辑
- 粘贴逻辑
- 空状态文案
- 加载状态组件
- `Search / Editor / Settings` 视觉

## 8. 影响文件

本次改动应主要落在：

- `src/features/picker/PickerShell.tsx`

必要时可少量调整：

- `src/index.css`

原则上不新增复杂共享组件，也不扩大改动到其他窗口。

## 9. 验证策略

### 9.1 构建验证

前端改动后至少执行：

`./scripts/win-pnpm build`

### 9.2 视觉验证

重点检查以下场景：

- 打开 `Picker` 后，是否能在 1 秒内判断当前选中项
- 使用上下键切换时，选中项是否始终是唯一明显主角
- 文本、图片、文件混排时，层级是否仍然清楚
- 收藏项在未选中状态下是否仍可识别，但不抢正文
- 顶部状态是否不会压过主内容区

### 9.3 交互回归

至少手动确认：

- `Enter` 粘贴
- `Ctrl+Enter` 打开编辑
- 收藏切换
- 数字键选择
- 鼠标单击切换选中项
- 双击粘贴

## 10. 验收标准

- 打开 `Picker` 后，当前选中项成为唯一明确视觉中心
- 未选中项整体退后，但依然可快速扫读
- 无需展开内容，也能明显建立“当前锁定项”感
- 数字键徽标与选中态的关系比当前更明确
- 顶部区域不再与主列表争夺视觉中心
- 不引入新的交互复杂度

## 11. 实施摘要

本次迭代将 `Picker` 从“多条风格接近的列表卡片”调整为“有明确当前锁定槽的专业工具面板”。

它不依赖新增信息，不依赖展开详情，也不依赖强化动作条，而是通过更精确的结构层级，让用户先看到“现在会贴哪条”。
