# 设置窗口布局优化设计

日期：2026-04-06

## 1. 背景

当前设置窗口已经从早期的 `Manager` 三栏管理界面收缩为独立的 `SettingsShell`，实现位置在 `src/features/settings/SettingsShell.tsx`。现状的优点是简单直接，但页面整体仍然更像“把设置表单按分组顺序堆起来”，而不是一个完成度高、结构稳定的桌面偏好设置窗口。

这带来几个感知层面的短板：

- 第一眼先看到的是一串控件，而不是清晰的结构骨架。
- 各分组虽然已经有标题，但更像线性长页的切段，缺少成熟桌面应用常见的导航感与定位感。
- `行为` 这类包含依赖关系的设置，视觉上与普通布尔开关权重接近，主次关系不够清楚。
- 页面整体气质仍偏“可用”，还没有进入“朴素但高级、经过打磨”的状态。

本次工作不解决功能缺失，而是解决设置窗口的结构完成度问题。

## 2. 目标

### 2.1 主要目标

- 让设置窗口第一眼先呈现结构，再呈现控件。
- 从“单页表单堆叠”提升为“有清晰骨架的桌面偏好设置页”。
- 保持整体气质克制、稳重，接近 GitHub Desktop 或系统偏好设置这类朴素但高级的产品感。
- 让用户能快速判断自己当前处于哪一组设置，并能稳定跳转到目标分组。

### 2.2 非目标

- 不新增设置项。
- 不改动设置模型、桥接层或 Tauri 命令层。
- 不把设置页改造成复杂的多页面路由系统。
- 不为了“设计感”引入与当前产品调性不匹配的强品牌化视觉语言。

## 3. 方案结论

本次采用：

`A：轻量双栏设置页`

即：

- 顶部保留简洁页头。
- 主体改为 `左侧分组导航 + 右侧内容区`。
- 仍然是同一窗口、同一页面、同一滚动上下文。
- 左侧导航负责定位与分组高亮，右侧负责呈现各分组的具体设置模块。

不采用：

- `B：继续保留单列长页，只增加顶部锚点`
- `C：真正的分栏偏好设置中心，一次只显示一个分组`
- `D：仪表板式卡片布局`

原因如下：

1. `A` 最符合用户选择的“朴素但高级”与“明确双栏骨架”。
2. `B` 改动最小，但提升更多停留在样式层，难以根治“像表单堆起来了”的问题。
3. `C` 的产品气质虽然更强，但对于当前设置项数量来说过重，会把这次工作从布局优化推向交互模型重写。
4. `D` 的展示感强于设置感，不适合当前窗口作为低频偏好设置页的角色。

## 4. 设计原则

### 4.1 先建立结构骨架，再优化控件观感

这次的第一目标不是换一套更漂亮的控件样式，而是让用户在进入窗口后的几秒内建立稳定认知：

- 这是设置窗口
- 这里有哪些分组
- 我现在位于哪一组

只有先建立这种结构感，页面才会从“表单堆叠”进入“成熟偏好设置页”的体验区间。

### 4.2 克制优先于花哨

设置页不是营销页面，也不是主工作界面。视觉上应优先追求安静、稳定、可信，而不是强风格化。布局、边界、间距、主次关系要比品牌动作更重要。

### 4.3 保持桌面应用的自然浏览方式

本次不引入真正的页面切换或复杂状态机。点击左侧分组只是帮助定位，右侧内容仍保持自然的文档式滚动体验，避免把一个低频设置页做成 IDE 后台。

### 4.4 用模块关系表达设置关系

不同设置之间的主次、依赖、归属关系，优先通过模块结构体现，而不是只依赖说明文字。尤其是“开机自启”与“静默启动”这类父子关系，应该在布局上自然呈现出来。

## 5. 布局骨架

### 5.1 顶部页头

页头继续保留当前设置窗口中的简洁感，但结构调整为更明确的“偏好设置”语义：

- 左侧显示 `FloatPaste` 与 `设置` 的标题组合。
- 标题下保留一行简短说明，强调“偏好设置会自动保存”。
- 右上角只保留轻量级保存状态，例如 `保存中 / 已保存 / 保存失败`，不引入醒目的通知式反馈。

页头职责是建立窗口身份，而不是承担装饰任务，因此不加入复杂操作或额外说明块。

### 5.2 主体双栏

主体区域改为双栏布局：

- 左栏：固定宽度的分组导航。
- 右栏：主要内容区。

左栏宽度应足以容纳一级分组文案，但不能宽到形成第二个内容区。右栏需要获得明显比当前更宽松的表单承载空间，因此窗口整体宽度也需要相应抬高。

相较当前 `max-width: 680px` 的单列设置页，本次建议将整体最大宽度提升到更适合双栏阅读的区间，大致控制在 `860px - 920px` 这一档。具体值在实现时可以根据真实观感微调，但目标是不让导航与内容互相挤压。

### 5.3 右侧仍为同页滚动

虽然主体是双栏，但页面不切成两个独立滚动容器：

- 用户看到的仍然是同一个设置页。
- 右侧内容按文档式顺序向下排列。
- 左侧仅作为始终可见的结构辅助，不成为第二个独立工作区。

这能保留自然的桌面应用感，同时避免双滚动区域带来的割裂与廉价感。

### 5.4 窄窗口退化策略

默认桌面运行态应优先保护双栏骨架，因此设置窗口需要同步提高最小可用宽度，避免用户把窗口缩到明显破坏布局的程度。

同时，为避免浏览器预览、异常尺寸或后续小窗口场景直接出现横向挤压，本次明确补充退化策略：

- 当窗口宽度处于双栏舒适区时，使用 `左侧导航 + 右侧内容`。
- 当可用宽度低于双栏下限时，不允许通过横向滚动硬保双栏。
- 低于阈值时自动退化为 `顶部简短分组锚点 + 单列内容区`。

也就是说，双栏是默认主方案，但窄窗退化采用“切回单列”，而不是“继续压缩双栏”或“制造横向滚动”。

为保证后续规划有统一口径，本次定义明确阈值：

- `>= 880px`：双栏布局。
- `< 880px`：退化为顶部锚点 + 单列布局。

同时约定：

- Tauri 桌面窗口的最小宽度目标值同步提升到 `880px`，尽量避免用户在真实桌面态进入退化布局。
- 浏览器预览、异常窗口尺寸、或未来小窗口场景允许进入退化布局，但这不改变主设计仍以桌面双栏为核心。

## 6. 左侧导航设计

### 6.1 导航内容

左侧导航只承载一级分组：

- 快捷键
- 通用
- 外观
- 行为
- 排除应用

不额外拆出二级目录，也不把每个设置项都变成导航项，避免左栏从“结构骨架”膨胀为“设置索引树”。

### 6.2 视觉形态

导航项应接近“目录”而不是“主应用菜单”：

- 不强依赖图标。
- 选中态使用低对比背景、轻边框，或非常克制的侧边强调条。
- 默认态以清晰可读为主，不制造强烈按钮感。

左栏顶部可放一个极轻的 `偏好设置` 标签，帮助建立上下文，但不能让左栏演变成说明面板。

### 6.3 行为

左侧导航承担两类行为：

1. 点击分组，右侧平滑滚动到对应模块。
2. 用户滚动右侧内容时，左侧当前分组自动同步高亮。

这样用户无论是“直接跳过去”还是“顺着看下去”，都不会丢失方位感。

为了避免实现口径不一致，本次额外约定 active section 规则：

- 以右侧内容区顶部下方的一条“观察带”作为判定基准，而不是简单取当前屏幕中面积最大的分组。
- 当某个分组标题进入该观察带时，将该分组视为当前 active。
- 多个分组同时满足条件时，优先选择更靠上的那个分组。
- 用户点击左侧导航触发平滑滚动时，在滚动完成前短暂锁定目标分组，避免高亮在滚动过程中来回跳动。

观察带的精确像素值可以在实现时根据真实页头高度微调，但交互原则必须固定为“靠近内容区上缘的分组为当前分组”。

为了让实现与测试可以直接对齐，再补充一层执行口径：

- 默认观察带以内容滚动容器顶部下方 `96px` 为基准线。
- 某个分组标题的顶部到达或越过该基准线时，该分组成为 active。
- 若多个分组标题都已越过基准线，则取最接近基准线的那个分组。
- 点击导航后，目标分组滚动停靠位置也以 `96px` 为补偿量，保证标题不会贴顶。
- 编程滚动期间锁定 active 目标，直到滚动结束并连续 `120ms` 无新滚动事件后，再恢复滚动驱动的 active 计算。

键盘与可访问性规则同步明确：

- 左侧分组容器使用 `nav` 语义并提供 `aria-label="设置分组"`。
- 导航项使用原生 `button` 或等价可聚焦控件，不实现自定义 roving tabindex。
- Tab 键按文档顺序聚焦各导航项，`Enter` 或 `Space` 触发滚动定位。
- 当前 active 项设置 `aria-current="true"`。
- 窄窗退化为顶部锚点时，沿用同一组导航语义，不切换成另一套交互模型。

## 7. 右侧内容区设计

### 7.1 内容组织

右侧不再是“section 标题 + 一串控件”的线性堆叠，而是“分组标题 + 分组说明 + 若干设置模块”的结构。

每个一级分组至少包含：

- 分组标题
- 一句非常简短的组级说明
- 1 到 2 个内部设置模块

这样右侧每一组都像一个自洽的配置单元，而不是长页上的切片。

### 7.2 快捷键

`快捷键` 分组拆成两个明确模块：

- 速贴唤起
- 搜索窗口

其中“搜索窗口”模块内部保留启用开关与快捷键输入框，但通过同一模块内的布局关系表达它们的关联，避免像现在一样只是“标题右侧塞一个启用勾选，再接一个输入框”。

为避免后续实现拆散，明确其状态边界如下：

- `搜索窗口` 是一个复合设置模块，而不是两个互不相关的设置行。
- 父状态为 `searchShortcutEnabled`。
- 子状态为 `searchShortcut`。
- 当父状态关闭时，子输入框保留当前值但进入禁用态，不清空内容。
- 该模块的所有状态仍由 `SettingsShell` 顶层持有，子组件只负责展示与事件回调，不自行维护持久状态。

### 7.3 通用

`通用` 分组保留数值配置，但要弱化“裸输入框”感：

- 历史记录上限
- 速贴窗口记录数

每项作为单独设置行或小模块呈现，让标签、输入区和说明文字形成稳定节奏，而不是纯粹靠垂直间距隔开。

### 7.4 外观

`外观` 分组保留现有的卡片式 radio 思路，但要收紧为更像桌面设置页的表达：

- 界面主题
- 速贴窗口显示位置

两类单选项仍可使用可点击卡片，但卡片尺寸、边界和留白应更克制，重点是整齐、稳定、可靠，而不是强调“卡片设计感”。

### 7.5 行为

`行为` 分组是本次层级优化的重点：

- 开机自启
- 开机时静默启动
- 回贴后恢复剪贴板
- 暂停监听

其中“开机时静默启动”不应继续被呈现为与其他布尔项完全同权的平铺行。它应该成为“开机自启”的从属设置，在结构上体现依赖关系，例如放入同一模块中作为次级配置，或作为缩进的下级项。

其余独立行为项则保持并列，但每项都应具备清晰的标题与必要说明，避免四个 checkbox 看起来像一排机械开关。

该复合模块的边界也需要提前固定：

- `开机自启` 是父状态，对应 `launchOnStartup`。
- `开机时静默启动` 是子状态，对应 `silentOnStartup`。
- 当父状态关闭时，子状态在数据上继续遵循现有逻辑：自动变为 `false`。
- 在 UI 上，子项保留可见，但以次级禁用态呈现，明确表达“它属于父项，但当前不可用”。
- 该父子模块同样由 `SettingsShell` 顶层统一持有状态，局部组件只接收受控值和事件处理器。

### 7.6 排除应用

`排除应用` 作为大文本输入区，应单独成为一个安静且完整的模块：

- 标题与说明明确。
- 输入区获得比普通输入框更充足的呼吸感。
- 不与小型布尔项或数值项混排。

这能避免页面底部再次回到“堆控件”的观感。

## 8. 交互与状态设计

### 8.1 滚动与定位

交互采用“同页滚动 + 左栏联动”的轻量方案：

- 点击左侧导航后，右侧滚动到目标分组。
- 右侧滚动时，左侧高亮跟随当前可见分组变化。
- 每个分组标题保留适度滚动补偿，避免滚动结束后标题贴边过紧。

### 8.2 左栏可见性

左侧导航应在滚动过程中持续可见，优先采用桌面设置页常见的“固定可见”或“sticky 可见”方式。

目的不是强化交互感，而是确保页面结构骨架始终存在。

### 8.3 自动保存

自动保存机制保持不变，继续沿用当前的 debounce 与状态提示逻辑：

- 不恢复显式“保存”按钮。
- 不把本次改动扩展成保存策略重写。
- 只调整保存状态在布局中的位置与权重。

### 8.4 错误态与禁用态

现有的错误提示、禁用逻辑继续沿用，但在新布局中要重新校正层级：

- 保存错误仍位于页头下方的高可见区域。
- 禁用项在视觉上仍保持清楚，但不至于破坏模块整体节奏。
- 依赖关系型禁用态要和模块结构保持一致，例如“静默启动”在父项关闭时仍然是同一模块中的次级禁用状态。

另外需要补齐首次加载失败的口径：

- 如果 `getSettings` 初次加载失败，页面不应直接用默认值渲染完整可编辑表单。
- 此时右侧内容区显示阻断式错误状态，明确告知“设置加载失败”。
- 错误状态中提供重试操作，允许重新请求设置。
- 只有在成功拿到服务端设置后，才进入正常的可编辑布局。

这样可以避免用户在未知真实配置的前提下修改一份“猜出来的本地默认值”，也让“加载失败”和“保存失败”形成完整但分层不同的错误闭环。

首次加载失败时的界面边界进一步明确如下：

- 页头仍然显示，用于保留窗口身份与上下文。
- 左侧导航不显示可交互分组列表，避免用户误以为可以进入尚未加载完成的内容区。
- 主内容区显示阻断式错误卡片，其中包含错误说明与主重试按钮。
- 连续重试失败时，页面保持在同一阻断态，仅更新错误信息，不切换为默认值表单。
- 一旦重试成功，再进入正常布局并恢复左侧导航与右侧内容。

## 9. 实现边界

### 9.1 不改功能范围

本次只做布局与信息层级优化，不新增或删除设置项：

- `shortcut`
- `searchShortcut`
- `searchShortcutEnabled`
- `historyLimit`
- `pickerRecordLimit`
- `pickerPositionMode`
- `restoreClipboardAfterPaste`
- `pauseMonitoring`
- `themeMode`
- `excludedApps`
- `launchOnStartup`
- `silentOnStartup`

这些字段的含义、数据流、保存逻辑都保持不变。

### 9.2 不引入真正路由

左侧导航不等同于页面切换，不会引入：

- 分组级路由状态
- 每组单独页面
- 左右栏完全独立的生命周期

本次只需要“定位”和“高亮”，不需要“页面系统”。

同时，窄窗口退化为顶部锚点时，也只是导航形态变化，不意味着进入另一套页面逻辑。

### 9.3 组件拆分保持克制

如果实现中需要拆出局部组件，应优先考虑少量薄组件，例如：

- `SettingsNav`
- `SettingsSection`
- `SettingGroupCard`
- `SettingRow`

不建议为了形式上的“整洁”把当前设置页过度拆碎成大量原子组件，否则会增加理解和维护成本。

## 10. 验证方案

### 10.1 构建验证

前端布局修改后至少执行：

`./scripts/win-pnpm build`

### 10.2 手动验证

重点验证以下场景：

1. 打开设置窗口后，第一眼是否先读到结构，再读到控件。
2. 点击左侧任一分组，右侧是否稳定滚动到正确位置。
3. 手动滚动右侧内容时，左侧当前分组高亮是否稳定，不出现乱跳。
4. 自动保存状态在新页头中是否清楚但不过度抢眼。
5. “开机自启 / 静默启动”的从属关系是否比当前更容易理解。
6. 错误提示、禁用态与文本域在新布局中是否仍然清晰。

### 10.3 最小状态矩阵

为避免规划阶段遗漏状态，本次至少覆盖以下界面状态：

| 状态 | 页头 | 左侧导航 | 右侧内容 | 备注 |
|------|------|----------|----------|------|
| 初始加载中 | 可见，弱化状态提示 | 不显示交互导航 | Loading 状态 | 不渲染可编辑表单 |
| 初始加载失败 | 可见 | 不显示交互导航 | 阻断式错误卡片 + 重试按钮 | 不使用本地默认值顶替 |
| 正常编辑 | 可见 | 正常显示并联动高亮 | 双栏或单列退化后的正常表单 | 主状态 |
| 保存中 | 可见，显示轻量保存中提示 | 正常显示 | 表单保持可见 | 不弹全局通知 |
| 保存失败 | 可见，显示错误反馈 | 正常显示 | 表单保持可见 | 允许继续调整或再次触发保存 |
| 窄窗退化 | 可见 | 退化为顶部锚点 | 单列内容区 | 低于 `880px` 生效 |

### 10.4 验收标准

- 设置窗口的第一印象不再是长页表单，而是有明确骨架的桌面偏好设置页。
- 左侧导航真实帮助定位，而不是装饰性的摆设。
- 右侧每个分组内部存在明确主次关系，不再所有控件同一重量。
- 页面气质整体克制、稳定、可信，不引入不必要的品牌化表演。
- 功能逻辑与自动保存行为保持原样，不因为布局优化引入新的交互复杂度。

## 11. 风险与后续

### 11.1 主要风险

本次的主要风险不是功能错误，而是双栏做得太轻或太重：

- 太轻：看起来只是“左边多了一列目录”，仍然没有成熟感。
- 太重：看起来像 IDE 设置后台，超出了当前产品需要的克制程度。

### 11.2 风险控制

为控制这个风险，本次实现需坚持以下边界：

- 左侧仅做一级分组，不做复杂树状结构。
- 右侧仍保持文档式连续阅读，不切成多个页面。
- 视觉上优先稳重、秩序和留白，不靠强图标、强色块或强装饰建立完成度。

### 11.3 后续可演进方向

如果这次双栏骨架落地后依然存在明显体验问题，再考虑下一轮单独设计：

- 顶部全局搜索或过滤设置项
- 更细粒度的分组摘要
- 移动或小尺寸窗口下的折叠导航方案

这些都不属于本轮范围。

## 12. 实施摘要

本次设置窗口布局优化将 `SettingsShell` 从“单列长页设置表单”升级为“轻量双栏桌面偏好设置页”。

其核心不是换一套更花的样式，而是通过 `页头 + 左侧分组导航 + 右侧模块化内容区` 的骨架，建立稳定、克制、成熟的结构感。同时保持自动保存机制、设置字段和数据流全部不变，把工作边界严格收在前端布局与信息层级优化之内。
