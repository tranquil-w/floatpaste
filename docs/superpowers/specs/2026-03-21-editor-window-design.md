# 独立编辑窗口设计

## 1. 背景

当前 `Workbench` 在“搜索/选中”和“文本编辑”两种任务之间共用同一窗口与同一套会话快捷键，导致以下交互冲突：

- 进入编辑状态后，`Enter` 无法作为文本换行键稳定工作。
- 进入编辑状态后，`ArrowUp / ArrowDown` 仍可能被解释为列表导航，而不是文本光标移动。
- 当前还存在 `Enter` 无法完成粘贴的缺陷，说明“确认动作”和“编辑动作”耦合在同一层后，快捷键语义已经变得脆弱。

从现有仓库结构和文档约束来看，`Picker` 的定位一直是“轻量选择 + 快速回贴”，复杂编辑并不适合继续堆叠在它的会话模型里。当前仓库也已经引入了独立的 `workbench` 窗口，这说明“按任务拆分窗口职责”已经是项目可接受的架构方向。

本次设计的核心目标不是继续给现有编辑态打补丁，而是从交互模型上拆开“选择”和“编辑”，彻底消除键位语义冲突。

## 2. 本轮目标

本轮设计目标如下：

1. 解决编辑态下 `Enter`、方向键与列表导航/确认动作冲突的问题。
2. 为文本编辑提供一个纯编辑语义的独立窗口，不承担粘贴职责。
3. 同时支持从 `Picker` 和 `Workbench` 搜索结果进入编辑，但一次只编辑当前项。
4. 关闭编辑窗口后，能够回到原来源窗口，并尽量保留来源窗口状态。
5. 为后续实现提供清晰的窗口职责边界、事件流和验证清单，避免再次出现快捷键残留类问题。

## 3. 已确认的设计决策

用户已经确认以下设计约束，这些约束视为本次实现的固定边界：

- 方案方向：新增独立的小型编辑窗口，而不是继续在现有 `Workbench` 内混合编辑态。
- 编辑入口范围：同时支持从 `Picker` 和 `Workbench` 搜索结果打开。
- 编辑职责：仅承担单条记录的纯编辑功能，不承担粘贴功能。
- 未保存关闭策略：关闭时弹出 `保存 / 放弃 / 取消关闭` 确认。
- 关闭返回策略：从哪里来就回哪里；从 `Picker` 来回 `Picker`，从 `Workbench` 来回 `Workbench`。

## 4. 备选方案对比

### 4.1 方案 A：继续在现有 `Workbench` 内做模式切换

思路是保留当前单窗口模型，在编辑态下动态屏蔽 `Up / Down / Enter` 对列表导航和确认动作的影响。

优点：

- 改动窗口结构最少。
- 现有组件与状态模型可继续复用。

缺点：

- 需要在 Rust 会话快捷键层和前端焦点层同时做条件屏蔽，复杂度高。
- 焦点丢失、弹窗出现、搜索框与编辑框切换时都容易再出边界 bug。
- 本轮问题本身就说明这条路已经产生明显副作用。

### 4.2 方案 B：新增独立单项编辑窗口

思路是新增专用 `editor` 窗口，把“列表导航”和“文本编辑”拆到不同窗口中。

优点：

- 职责边界最清楚，键位冲突从模型层消失。
- 编辑窗口内 `Enter`、方向键天然回归文本编辑语义。
- `Picker`、`Workbench`、`Editor` 三者可分别维护自己的快捷键模型。
- 更符合当前仓库已经采用多窗口职责拆分的方向。

缺点：

- 需要新增一个窗口会话、事件和返回流程。
- 需要处理来源窗口状态恢复。

### 4.3 方案 C：保留单窗口，但重做整体快捷键体系

思路是把搜索/列表导航改为别的快捷键组合，避免与文本编辑键位冲突。

优点：

- 理论上可以减少新窗口数量。

缺点：

- 会让现有直觉交互变重。
- `Picker` 与 `Workbench` 的键盘心智模型会进一步分裂。
- 不能从根本上解决“一个窗口承担两类任务”带来的复杂度。

### 4.4 结论

本次推荐采用 **方案 B：新增独立单项编辑窗口**。

原因不是单纯“多开一个窗口”，而是它能把三种不同任务拆成三种清晰职责：

- `Picker`：快速选择
- `Workbench`：搜索与定位
- `Editor`：单项编辑

这条路线能够让键盘行为重新回到任务本身，而不是靠焦点判断和快捷键屏蔽补丁维持。

## 5. 设计原则

### 5.1 任务单一优先于模式叠加

一个窗口只承担一种主要任务。只要某个窗口同时承担“列表导航”和“文本编辑”，就需要在同一组键位上做语义切换，长期一定会积累复杂度。

### 5.2 快捷键必须服务当前任务，而不是服务全局妥协

在 `Editor` 中，键盘首先应该像一个可靠的文本编辑器，而不是一个带文本框的列表窗口。因此 `Enter`、方向键、光标移动必须完全回归文本语义。

### 5.3 来源上下文需要可恢复，但不应侵入编辑窗职责

编辑窗需要知道“从哪里来、回哪里去”，但不负责重新承载来源窗口的列表与搜索逻辑。来源上下文只作为打开和关闭时的会话元信息存在。

### 5.4 保护用户修改优先于关闭速度

纯编辑窗口必须把未保存保护放在一等公民位置。只要用户做了修改，任何关闭路径都必须统一进入确认流程。

## 6. 窗口职责拆分

### 6.1 Picker

`Picker` 保持轻量职责：

- 展示最近记录
- 上下选择当前项
- 触发回贴
- 打开 `Workbench` 搜索
- 打开 `Editor` 编辑当前项

`Picker` 不再承载文本编辑行为，也不再需要为编辑态重写 `Enter` 或方向键语义。

### 6.2 Workbench

`Workbench` 保持“搜索与定位”职责：

- 搜索结果列表
- 选中某条记录
- 查看摘要或详情信息
- 从当前选中项打开 `Editor`

`Workbench` 不再承担深度文本编辑职责。若未来保留详情展示，也应明确区分“只读详情”和“进入编辑器”。

### 6.3 Editor

新增 `Editor` 作为纯编辑窗口：

- 加载单条文本记录
- 提供文本编辑能力
- 保存当前修改
- 关闭并返回来源窗口
- 处理未保存确认

`Editor` 不承担以下职责：

- 列表导航
- 搜索
- 粘贴
- 编辑上一条/下一条记录

## 7. 编辑会话模型

建议为 `Editor` 定义独立会话载荷，至少包含以下信息：

- `itemId`：当前编辑的记录 id
- `source`：来源窗口，值为 `picker` 或 `workbench`
- `returnTo`：关闭后应恢复的窗口，值为 `picker` 或 `workbench`

必要时可再补充：

- `sourceSelectionId`：来源窗口当前选中项 id
- `sourceKeyword`：来源为 `Workbench` 时的当前搜索词

这些信息仅用于来源状态恢复与窗口协调，不用于把 `Editor` 变成新的列表容器。

## 8. 事件流设计

### 8.1 从 Picker 打开 Editor

推荐流程如下：

1. 用户在 `Picker` 中选中某条记录。
2. 用户触发“编辑当前项”动作。
3. Rust 侧先结束 `Picker` 会话快捷键。
4. `WindowCoordinator` 打开 `editor` 窗口，并发送 `editor://session-start`。
5. `Editor` 根据 `itemId` 加载详情并自动聚焦编辑框。

### 8.2 从 Workbench 打开 Editor

推荐流程如下：

1. 用户在 `Workbench` 搜索结果中定位到目标记录。
2. 用户触发“编辑当前项”动作。
3. Rust 侧暂停或释放 `Workbench` 的会话快捷键。
4. `WindowCoordinator` 打开 `editor` 窗口，并发送 `editor://session-start`。
5. `Editor` 加载当前项并自动聚焦编辑框。

### 8.3 关闭 Editor 并返回来源窗口

推荐流程如下：

1. `Editor` 收到关闭请求。
2. 若无未保存修改，则直接关闭。
3. 若存在未保存修改，则弹出统一确认框。
4. 关闭成功后，Rust 侧依据 `returnTo` 恢复对应来源窗口。
5. 来源窗口重新获得会话快捷键。

这个流程的关键点是：**来源窗口快捷键的释放与恢复由窗口协调层统一处理，不由前端分散拼接。**

## 9. 编辑窗口交互设计

### 9.1 页面结构

编辑窗口建议保持克制，不做成新的“管理页”：

- 顶部：来源说明、当前条目基础信息、关闭入口
- 中部：主文本编辑区
- 底部：`保存`、`关闭` 两个主动作

可选补充：

- 创建时间
- 来源应用
- 类型信息

这些信息只能做辅助，不应压过主编辑区。

### 9.2 键盘规则

编辑窗口内的键盘规则应固定如下：

- `Enter`：换行
- `ArrowUp / ArrowDown / ArrowLeft / ArrowRight`：文本光标移动
- `Ctrl+S`：保存
- `Esc`：请求关闭

明确不支持：

- `Enter` 触发保存
- `Enter` 触发粘贴
- `ArrowUp / ArrowDown` 切换上一条或下一条记录

### 9.3 保存策略

保存行为建议保持单纯：

- 只更新当前记录文本内容
- 保存成功后更新 `savedText`
- 保存成功后清除脏状态
- 保存成功后通知相关查询失效，确保来源窗口再次显示时拿到最新文本
- 保存后默认不自动关闭

“保存”和“关闭”必须是两个独立动作，以保持编辑器心智稳定。

### 9.4 未保存确认

任何关闭路径都统一走同一套确认逻辑：

- `保存并关闭`
- `放弃修改`
- `取消`

需要覆盖的关闭入口包括：

- 顶部关闭按钮
- `Esc`
- 窗口系统关闭按钮

若未来存在其他跳转入口，也必须复用同一关闭策略。

## 10. 非文本记录处理

本次编辑窗只服务文本记录。

对非文本记录的处理建议如下：

- 在 `Picker` 和 `Workbench` 的入口层禁用“编辑”动作，避免误入。
- 如果由于状态不同步等异常原因进入 `Editor`，则展示明确提示“该类型暂不支持文本编辑”，并禁止保存。

入口层禁用优先，编辑窗兜底次之。

## 11. 快捷键与窗口协调设计

这是本次方案最关键的技术约束。

### 11.1 当前问题本质

现有问题不是 `textarea` 本身不支持换行，而是 `Workbench` 会话快捷键仍在更高层把 `Up / Down / Enter` 解释为导航与确认动作。因此只要编辑态仍在同一个会话模型内，这类冲突就会持续出现。

### 11.2 调整原则

推荐按窗口而不是按焦点去划分快捷键：

- `Picker` 会话快捷键只属于 `Picker`
- `Workbench` 会话快捷键只属于 `Workbench`
- `Editor` 不注册列表导航与确认类会话快捷键

换言之，`Editor` 打开时，不应存在仍在后台生效的 `Picker / Workbench` 会话级 `Up / Down / Enter` 拦截。

### 11.3 推荐做法

在窗口协调层统一管理：

1. 打开 `Editor` 前先释放来源窗口会话快捷键。
2. `Editor` 打开后，不注册与文本输入冲突的会话快捷键。
3. `Editor` 关闭后，再按 `returnTo` 恢复来源窗口快捷键。

这样能避免把“当前焦点在哪个 DOM 元素上”这种脆弱判断下放到前端交互层。

## 12. 数据与状态恢复要求

### 12.1 从 Picker 返回

从 `Picker` 打开编辑后关闭返回时，期望保留：

- 当前列表内容
- 当前选中项
- 当前窗口位置与大小

至少不能在关闭编辑后把 `Picker` 回退成一次全新会话。

### 12.2 从 Workbench 返回

从 `Workbench` 打开编辑后关闭返回时，期望保留：

- 当前搜索词
- 当前结果列表
- 当前选中项
- 当前滚动位置

若短期内难以完整保留滚动位置，最低要求也要保留搜索词和当前选中项，避免用户重新检索。

## 13. 范围界定

### 13.1 在范围内

- 新增 `editor` 窗口与前后端会话模型
- 从 `Picker` 打开当前项编辑
- 从 `Workbench` 搜索结果打开当前项编辑
- 编辑窗文本编辑、保存、关闭、未保存确认
- 来源窗口快捷键释放与恢复
- 相关查询刷新与来源状态恢复

### 13.2 不在范围内

- 在编辑窗内直接粘贴
- 在编辑窗内切换上一条/下一条记录
- 在编辑窗内做完整搜索和管理能力
- 为非文本记录新增专门编辑器
- 重做 `Picker` 的整体交互模型

## 14. 风险与防护

### 14.1 快捷键残留

风险：

- `Editor` 打开后，旧窗口的会话快捷键仍然生效，继续拦截 `Enter` 或方向键。

防护：

- 必须在 Rust 窗口协调层统一释放来源窗口会话快捷键。
- 关闭返回时再统一恢复，不允许前端自行拼接多步命令替代。

### 14.2 来源窗口状态丢失

风险：

- 从 `Workbench` 返回后丢失搜索词、选中项或列表上下文。

防护：

- 会话启动时显式传递来源信息。
- 来源窗口在被覆盖期间保留必要状态，不做无条件清空。

### 14.3 未保存关闭路径遗漏

风险：

- 顶部关闭按钮能拦截，但 `Esc` 或系统关闭按钮漏掉确认逻辑。

防护：

- 所有关闭入口收敛到同一个 `requestClose()` 流程。

### 14.4 非文本项误入编辑窗

风险：

- 用户进入一个并不支持编辑的条目，得到错误预期。

防护：

- 入口层禁用编辑动作。
- 编辑窗再次校验记录类型并提供兜底提示。

## 15. 验证清单

实现后至少验证以下场景：

1. `Picker -> Editor`
   能打开当前项，编辑框自动聚焦，`Enter` 可以换行，方向键只移动光标，关闭后回到 `Picker`。

2. `Workbench -> Editor`
   能从搜索结果打开当前项，关闭后回到原 `Workbench`，保留搜索词与当前项。

3. 未保存保护
   顶部关闭按钮、`Esc`、系统关闭按钮都能弹出 `保存 / 放弃 / 取消`。

4. 快捷键隔离
   `Editor` 内按 `Up / Down / Enter` 不会触发列表切换、回贴或确认动作。

5. 保存后数据同步
   保存后来源窗口重新看到的是更新后的文本。

6. 非文本项保护
   非文本记录不会进入可编辑态，或进入后只能看到不可编辑提示。

## 16. 推荐落地顺序

建议按以下顺序实施：

1. 新增 `editor` 窗口与会话事件模型。
2. 在 Rust 窗口协调层补齐打开、关闭、返回来源的统一流程。
3. 新建前端 `EditorShell`，完成文本加载、编辑、保存和未保存确认。
4. 在 `Picker` 中补“编辑当前项”入口。
5. 在 `Workbench` 搜索结果中补“编辑当前项”入口。
6. 补齐来源窗口状态恢复与数据刷新验证。

这个顺序优先先把结构边界立稳，再补交互细节，能最大限度减少返工。

## 17. 结论

本次不推荐继续在现有 `Workbench` 编辑态上打补丁，而是推荐新增独立 `Editor` 窗口，把选择、搜索、编辑三个任务拆开。

该方案的核心收益是：

- 文本编辑键位恢复自然语义
- 会话快捷键职责边界清晰
- `Picker` 与 `Workbench` 都能继续保持各自的轻量职责
- 未保存保护与来源返回路径更容易做对

这是一条初始改动略大、但长期维护成本明显更低的路线。
